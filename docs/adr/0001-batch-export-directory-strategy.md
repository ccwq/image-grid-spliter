# ADR-0001：批量导出的目录策略

## 状态

已确认（需求收敛中，尚未实施）

## 背景

图片裁切会产生多个文件。传统 `<a download>` 只能交由浏览器下载，无法指定本地目录；当浏览器启用“每次询问保存位置”时，会对每个文件展示确认。

## 决策

1. 在支持 `File System Access API` 的 Chromium 浏览器中，使用用户选择的 `FileSystemDirectoryHandle` 直接写入多个切片文件。
2. 首次写入前由用户选择目录；应用将目录句柄保存到 IndexedDB。后续导出先尝试复用该句柄，并在浏览器要求时重新取得用户授权。
3. 不支持、被拒绝或写入失败时，**不提供 ZIP 导出**，而是降级回现有逐张 `<a download>` 下载流程。
4. 目录内已存在同名切片时不得覆盖。应用在扩展名前追加 `-1`、`-2` 等递增后缀，例如 `photo-01.png` 变为 `photo-01-1.png`。
5. 现有“下载全部”是唯一的主导出入口：支持目录直写时，首次点击选择目录；已保存且仍获授权时直接写入。导出设置中提供“更换导出目录”操作。
6. 目录直写中途失败时立即停止，并展示已成功和未完成的文件数量。用户可明确选择将未完成项用传统逐张下载重试；应用不得自动混合两个保存位置。
7. 自动下载仅在已有有效目录写入权限时使用目录直写；首次选择目录不能由自动下载触发。没有有效目录权限时，自动下载保持现有传统逐张下载行为。

## 后果

- Chrome/Edge 用户通常只需首次选择输出目录，后续切片不会逐张弹出保存对话框。
- Firefox/Safari 用户保留当前体验；其浏览器下载设置决定是否逐张要求确认保存位置。
- 直接写入前需检查重名并生成不冲突文件名，导出耗时会随已有同名文件数量略增。
- 主流程不增加新的导出按钮；用户可通过导出设置主动更换目标目录。
- 中途失败不会静默改用浏览器默认下载目录；用户拥有是否继续以传统下载完成剩余文件的控制权。
- 自动下载不会主动弹出目录选择器，避免在非用户点击上下文调用受限浏览器能力。
- 产品会跨刷新和后续打开应用尝试复用目录，但不能承诺目录权限永久有效：重启、用户撤销授权或浏览器策略变化后，可能需要重新确认或选目录。
- 纯前端与 GitHub Pages 部署方式不变。
