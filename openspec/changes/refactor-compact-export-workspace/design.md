## Context

该工具已具备多图裁切、网格编辑、格式设置、目录直写、传统下载降级与 PhotoSwipe 预览。重构只改变信息架构和反馈，不改变业务入口或删除功能。

设计读法：高频工具型 PWA，保留深色像素风和青绿色强调色，设计方差 3、动效强度 2、视觉密度 8。布局以任务效率优先，不引入新的设计系统或视觉主题。

## Decisions

### 布局

- 桌面（>= 960px）：`AppHeader` 后使用两列 Grid。左列为固定宽度的紧凑控制流（网格、裁切参数、导出设置），右列为上传区或结果区。控制列在长结果列表时 sticky。
- 移动：单列连续页面。上传/结果在前，控制面板可折叠且不迁移到二级页面。
- 所有视口沿用同一圆角、间距和按钮高度规则，避免桌面宽松而移动紧凑的双重样式。

### 导出进度 module

`useImageSlicer` 暴露单个 `exportProgress` Interface：阶段、当前值、总数、导出方式和最终报告。内部实现管理生成、目录写入和传统下载的计数与自动收起计时器。`ExportProgressOverlay` 只根据该 Interface 渲染 fixed 顶部反馈，因此不理解 Blob、目录句柄或下载链接。

### 生命周期

1. 用户手动下载或自动下载开始时，若需要重新生成，显示“正在生成 current / total”。
2. 目录直写显示“正在保存 current / total”；传统路径显示“正在触发下载 current / total”。
3. 成功时切换为报告并在五秒后自动收起；用户可以主动关闭。
4. 部分失败保持报告，不自动收起，并继续显示传统下载重试入口。

### 报告语义

- 目录导出：已保存、重命名、失败/未完成数。
- 传统下载：只报告已触发给浏览器的文件数，绝不声称已保存到磁盘。

## Non-Goals

- 不更换色彩主题、字体系统或引入第三方 UI 库。
- 不把现有功能隐藏到新路由、Tab 或二级页面。
- 不绕过浏览器对传统下载或目录权限的安全限制。

## Risks

- 长结果列表配合 sticky 控制栏必须在窄桌面退化为单列。
- 进度更新不能阻塞 Canvas 裁切或逐文件下载。
- 重构后必须验证 PhotoSwipe、隐藏 file input ref 和目录权限入口仍然可用。
